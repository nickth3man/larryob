# Building a Basketball-Reference clone from scratch

Building a full Basketball-Reference.com clone requires a relational database with **20+ interconnected tables** spanning 80 years of NBA history, data pipelines pulling from at least five distinct sources, and a scraping layer that navigates Basketball-Reference's unique HTML quirks — comment-wrapped tables, `over_header` rows, and a strict 20-requests-per-minute rate limit. This guide covers every technical detail needed across schema design, data acquisition, and web scraping.

The core challenge is not complexity per se but **data completeness across eras**. Steals, blocks, and turnovers were not tracked until 1973–74. Offensive and defensive rebounds were not split until the same season. Play-by-play data only exists digitally from 1996–97. Player-tracking spatial data begins in 2013–14. Any schema must accommodate these gaps with nullable columns, and any data pipeline must stitch together multiple sources to cover the full 1946–present timeline.

---

## Part 1: Complete database schema and data model

The schema below uses PostgreSQL syntax. Every table includes the rationale for its design, with column names mapped to Basketball-Reference's standard abbreviations.

### Core entity tables

**Franchises and teams** require careful separation. A franchise represents the continuous organizational entity (e.g., franchise `OKC` includes the Seattle SuperSonics era), while a team record represents one franchise in one season. This is how Basketball-Reference models the SuperSonics → Thunder, Nets → Brooklyn, and other relocations.

```sql
-- Franchise: the continuous organizational entity across relocations/renames
CREATE TABLE franchise (
    franchise_id    VARCHAR(3) PRIMARY KEY,  -- e.g., 'OKC' (current abbreviation)
    current_name    VARCHAR(100) NOT NULL,
    founded_year    SMALLINT NOT NULL,        -- e.g., 1967 for SuperSonics/Thunder
    is_active       BOOLEAN DEFAULT TRUE
);

-- Team: one franchise in one season with a specific name/city
CREATE TABLE team_season (
    team_season_id  SERIAL PRIMARY KEY,
    franchise_id    VARCHAR(3) REFERENCES franchise(franchise_id),
    season          SMALLINT NOT NULL,        -- ending year, e.g., 2024 for 2023-24
    team_abbrev     VARCHAR(3) NOT NULL,      -- SEA before 2009, OKC after
    team_name       VARCHAR(100) NOT NULL,    -- 'Seattle SuperSonics' or 'Oklahoma City Thunder'
    city            VARCHAR(50),
    arena           VARCHAR(100),
    conference      VARCHAR(4),               -- 'East' or 'West'
    division        VARCHAR(20),
    wins            SMALLINT, losses SMALLINT,
    playoff_wins    SMALLINT, playoff_losses SMALLINT,
    finish          VARCHAR(50),              -- 'Won Finals', 'Lost Conf. Semis', etc.
    pace            NUMERIC(5,1),             -- possessions per 48 min
    ortg            NUMERIC(5,1),             -- offensive rating (pts per 100 poss)
    drtg            NUMERIC(5,1),             -- defensive rating
    net_rtg         NUMERIC(5,1),
    srs             NUMERIC(5,2),             -- Simple Rating System
    sos             NUMERIC(5,2),             -- Strength of Schedule
    expected_wins   NUMERIC(4,1),             -- Pythagorean W
    attendance      INTEGER,
    UNIQUE(franchise_id, season)
);
CREATE INDEX idx_team_season_season ON team_season(season);
```

**Players** store biographical data with the Basketball-Reference player ID as the natural key. Height is stored in inches (integer) for easy querying; display formatting is an application concern.

    college         VARCHAR(100),
    high_school     VARCHAR(100),
    draft_year      SMALLINT,
    draft_round     SMALLINT,
    draft_pick      SMALLINT,

### Statistical tables: the multi-view approach

Basketball-Reference presents the same underlying data in **six views**: Totals, Per Game, Per 36 Minutes, Per 100 Possessions, Advanced, and Shooting. The most efficient schema stores **raw totals** and computes derived views. This avoids data duplication and ensures consistency.

```sql
-- Player season stats: one row per player per team per season per season_type
-- This is the TOTALS table; per-game, per-36, per-100 are computed views
CREATE TABLE player_season_stats (
    id              SERIAL PRIMARY KEY,
    player_id       VARCHAR(20) REFERENCES player(player_id),
    season          SMALLINT NOT NULL,
    team_abbrev     VARCHAR(3) NOT NULL,
    season_type     VARCHAR(10) NOT NULL DEFAULT 'RS',  -- 'RS', 'PO', 'ASG'
    age             SMALLINT,
    lg              VARCHAR(3),               -- 'NBA', 'ABA', 'BAA'
    pos             VARCHAR(5),               -- 'PG', 'SG-SF', etc.
    games           SMALLINT,
    games_started   SMALLINT,                 -- NULL before 1981-82
    minutes         INTEGER,                  -- total minutes played
    -- Shooting totals
    fgm             INTEGER, fga INTEGER,     -- field goals made/attempted
    fg3m            INTEGER, fg3a INTEGER,    -- 3-pointers (NULL before 1979-80)
    ftm             INTEGER, fta INTEGER,     -- free throws
    -- Rebounds
    orb             INTEGER,                  -- NULL before 1973-74
    drb             INTEGER,                  -- NULL before 1973-74
    trb             INTEGER,                  -- total rebounds
    -- Other counting stats
    ast             INTEGER,
    stl             INTEGER,                  -- NULL before 1973-74
    blk             INTEGER,                  -- NULL before 1973-74
    tov             INTEGER,                  -- NULL before 1977-78
    pf              INTEGER,                  -- personal fouls
    pts             INTEGER,
    -- Plus-minus (only available in modern era)
    plus_minus      INTEGER,
    UNIQUE(player_id, season, team_abbrev, season_type)
);
CREATE INDEX idx_pss_season ON player_season_stats(season);
CREATE INDEX idx_pss_player ON player_season_stats(player_id);

-- Computed view: Per-Game stats
CREATE VIEW player_per_game AS
SELECT id, player_id, season, team_abbrev, season_type, age, pos, games, games_started,
    ROUND(minutes::NUMERIC / NULLIF(games, 0), 1) AS mpg,
    ROUND(fgm::NUMERIC / NULLIF(games, 0), 1) AS fgm_pg,
    ROUND(fga::NUMERIC / NULLIF(games, 0), 1) AS fga_pg,
    ROUND(fgm::NUMERIC / NULLIF(fga, 0), 3) AS fg_pct,
    ROUND(fg3m::NUMERIC / NULLIF(games, 0), 1) AS fg3m_pg,
    ROUND(fg3a::NUMERIC / NULLIF(games, 0), 1) AS fg3a_pg,
    ROUND(fg3m::NUMERIC / NULLIF(fg3a, 0), 3) AS fg3_pct,
    ROUND(ftm::NUMERIC / NULLIF(games, 0), 1) AS ftm_pg,
    ROUND(fta::NUMERIC / NULLIF(games, 0), 1) AS fta_pg,
    ROUND(ftm::NUMERIC / NULLIF(fta, 0), 3) AS ft_pct,
    ROUND(orb::NUMERIC / NULLIF(games, 0), 1) AS orb_pg,
    ROUND(drb::NUMERIC / NULLIF(games, 0), 1) AS drb_pg,
    ROUND(trb::NUMERIC / NULLIF(games, 0), 1) AS trb_pg,
    ROUND(ast::NUMERIC / NULLIF(games, 0), 1) AS ast_pg,
    ROUND(stl::NUMERIC / NULLIF(games, 0), 1) AS stl_pg,
    ROUND(blk::NUMERIC / NULLIF(games, 0), 1) AS blk_pg,
    ROUND(tov::NUMERIC / NULLIF(games, 0), 1) AS tov_pg,
    ROUND(pf::NUMERIC / NULLIF(games, 0), 1) AS pf_pg,
    ROUND(pts::NUMERIC / NULLIF(games, 0), 1) AS ppg
FROM player_season_stats;

-- Computed view: Per-36 Minutes
CREATE VIEW player_per_36 AS
SELECT id, player_id, season, team_abbrev, season_type, age, pos, games,
    ROUND(fgm::NUMERIC * 36 / NULLIF(minutes, 0), 1) AS fgm_p36,
    ROUND(fga::NUMERIC * 36 / NULLIF(minutes, 0), 1) AS fga_p36,
    ROUND(pts::NUMERIC * 36 / NULLIF(minutes, 0), 1) AS pts_p36,
    ROUND(trb::NUMERIC * 36 / NULLIF(minutes, 0), 1) AS trb_p36,
    ROUND(ast::NUMERIC * 36 / NULLIF(minutes, 0), 1) AS ast_p36
    -- ... same pattern for all counting stats
FROM player_season_stats;
```

**Advanced stats** require their own table because several metrics (PER, WS, BPM, VORP) depend on league-wide averages and team-level data — they cannot be computed from a single player's box score alone.

```sql
CREATE TABLE player_advanced_stats (

    player_id       VARCHAR(20) REFERENCES player(player_id),
    season          SMALLINT NOT NULL,
    team_abbrev     VARCHAR(3),
    season_type     VARCHAR(10) DEFAULT 'RS',
    -- Efficiency
    per             NUMERIC(4,1),             -- Player Efficiency Rating
    ts_pct          NUMERIC(4,3),             -- True Shooting %
    efg_pct         NUMERIC(4,3),             -- Effective FG%
    -- Rate stats
    three_par       NUMERIC(4,3),             -- 3P Attempt Rate (3PA/FGA)
    ftr             NUMERIC(4,3),             -- Free Throw Rate (FTA/FGA)
    -- Percentage stats (estimates of % of available X while on floor)
    orb_pct         NUMERIC(4,1),
    drb_pct         NUMERIC(4,1),
    trb_pct         NUMERIC(4,1),
    ast_pct         NUMERIC(4,1),
    stl_pct         NUMERIC(4,1),
    blk_pct         NUMERIC(4,1),
    tov_pct         NUMERIC(4,1),
    usg_pct         NUMERIC(4,1),             -- Usage Rate
    -- Win Shares
    ows             NUMERIC(5,1),             -- Offensive Win Shares
    dws             NUMERIC(5,1),             -- Defensive Win Shares
    ws              NUMERIC(5,1),             -- Total Win Shares
    ws_per_48       NUMERIC(5,3),             -- Win Shares per 48 min
    -- Box Plus-Minus
    obpm            NUMERIC(4,1),             -- Offensive BPM
    dbpm            NUMERIC(4,1),             -- Defensive BPM
    bpm             NUMERIC(4,1),             -- Total BPM
    vorp            NUMERIC(5,1),             -- Value Over Replacement Player
    UNIQUE(player_id, season, team_abbrev, season_type)
);
```

### How to handle players on multiple teams in one season

Basketball-Reference creates a **TOT** (total) row plus individual rows for each team stint. The schema handles this naturally: a player who played for three teams in 2024 gets three rows in `player_season_stats` (one per `team_abbrev`) plus a fourth where `team_abbrev = 'TOT'`. The `TOT` row is computed, not a separate source. A useful approach is to compute it via a materialized view or insert it during ETL:

```sql
-- Insert TOT rows for multi-team players
INSERT INTO player_season_stats (player_id, season, team_abbrev, season_type, ...)
SELECT player_id, season, 'TOT', season_type,
    MAX(age), STRING_AGG(DISTINCT pos, '-'), SUM(games), SUM(games_started),
    SUM(minutes), SUM(fgm), SUM(fga), SUM(fg3m), SUM(fg3a),
    SUM(ftm), SUM(fta), SUM(orb), SUM(drb), SUM(trb),
    SUM(ast), SUM(stl), SUM(blk), SUM(tov), SUM(pf), SUM(pts)
FROM player_season_stats
WHERE season_type = 'RS'
GROUP BY player_id, season, season_type
HAVING COUNT(DISTINCT team_abbrev) > 1;
```

### Handling historical data gaps

The nullable columns in the schema already accommodate missing data. Additional considerations: **games started** is NULL before 1981–82. **Three-point fields** are NULL before 1979–80 (NBA) or 1967–68 (ABA). The `league_season` table's `three_point_line` flag helps queries adapt. For the pre-digital era, game-level play-by-play data does not exist; the `play_by_play` table simply has no rows for those games.

## Part 2: Data sources and APIs for every data type

### The NBA.com Stats API is your primary free source

The unofficial JSON API at **`https://stats.nba.com/stats/`** powers NBA.com's own website and is the single most comprehensive free data source. It returns JSON with `resultSets` arrays containing `headers` and `rowSet` data. Coverage varies by endpoint but extends back to **1946** for basic stats.

**Critical requirement**: All requests must include browser-like headers — particularly the `Referer: https://stats.nba.com/` header. Without it, the API returns 403 errors. Cloud-hosted IPs (AWS, GCP, Heroku, DigitalOcean) are **blocked entirely**; requests must originate from residential IPs or use residential proxy services.

Key endpoints and what they provide:

| Endpoint | Data returned | Coverage |
|---|---|---|
| `leaguedashplayerstats` | All player stats for a season with 30+ filters | 1946–present |
| `playergamelog` | Game-by-game stats for one player | 1946–present |
| `shotchartdetail` | Shot x,y coordinates, zones, distances | 1996–present |
| `playbyplayv2` | Full play-by-play event log per game | 1996–present |
| `commonplayerinfo` | Player bio, draft info, height, weight, school | All players |
| `boxscoreadvancedv2` | Advanced box score (ORtg, DRtg, TS%, pace, PIE) | Per game |
| `leaguestandings` | Conference/division standings | Historical |
| `drafthistory` | Complete draft history | All years |
| `leaguedashteamstats` | Team aggregate stats with measure type filters | Historical |
| `leaguegamefinder` | Search games by any criteria | Historical |
| `boxscoretraditionalv2` | Standard box score for a game | Per game |

Safe request pacing is approximately **1 request per second**, with a 600ms minimum. No official documentation exists — endpoints can change without notice.

### The nba_api Python library wraps every endpoint

The `nba_api` library (v1.11.4, actively maintained, MIT license) wraps **100+ endpoint classes** in a clean Python interface. Install with `pip install nba_api`. It provides three module families: `nba_api.stats.endpoints` (historical stats), `nba_api.stats.static` (embedded player/team lookups requiring no HTTP), and `nba_api.live.nba.endpoints` (live game data).

```python
from nba_api.stats.static import players, teams
from nba_api.stats.endpoints import (
    PlayerCareerStats, PlayerGameLog, ShotChartDetail,
    PlayByPlayV2, CommonPlayerInfo, LeagueDashPlayerStats,
    BoxScoreTraditionalV2, DraftHistory, LeagueStandings
)
import time

# Static lookups — no HTTP call needed
lebron = [p for p in players.get_players() if p['full_name'] == 'LeBron James'][0]
lakers = [t for t in teams.get_teams() if t['abbreviation'] == 'LAL'][0]

# Career stats (returns multiple DataFrames: regular season, post-season, career)
career = PlayerCareerStats(player_id=lebron['id'])
reg_season_df = career.get_data_frames()[0]

# Season-wide player stats with 30+ filter parameters
time.sleep(1)
league_stats = LeagueDashPlayerStats(
    season='2024-25',
    season_type_all_star='Regular Season',
    per_mode_detailed='PerGame',
    measure_type_detailed_defense='Base'
)
all_players_df = league_stats.get_data_frames()[0]

# Shot chart with court coordinates (available 1996-97+)
time.sleep(1)
shots = ShotChartDetail(
    team_id=0, player_id=lebron['id'],
    context_measure_simple='FGA',
    season_nullable='2024-25',
    season_type_all_star='Regular Season'
)
shot_df = shots.get_data_frames()[0]   # columns: LOC_X, LOC_Y, SHOT_MADE_FLAG, etc.

# Play-by-play for a specific game
time.sleep(1)
pbp = PlayByPlayV2(game_id='0022400100')
pbp_df = pbp.get_data_frames()[0]
# Columns: EVENTNUM, EVENTMSGTYPE, PERIOD, PCTIMESTRING, HOMEDESCRIPTION,
#           VISITORDESCRIPTION, SCORE, SCOREMARGIN, PLAYER1_ID, PLAYER1_NAME, etc.
```

A robust retry wrapper is essential for bulk data collection:

```python
from requests.exceptions import ReadTimeout, ConnectionError
import time

def safe_nba_api_call(endpoint_class, max_retries=3, delay=1.5, **kwargs):
    """Call nba_api endpoint with exponential backoff and retry logic."""
    for attempt in range(max_retries):
        try:
            time.sleep(delay)
            result = endpoint_class(**kwargs, timeout=60)
            return result.get_data_frames()
        except (ReadTimeout, ConnectionError) as e:
            wait = delay * (2 ** attempt)
            print(f"Attempt {attempt+1} failed: {e}. Retrying in {wait}s...")
            time.sleep(wait)
    raise Exception(f"Failed after {max_retries} attempts")
```

### Balldontlie API offers structured access at low cost

The **balldontlie API** (`https://api.balldontlie.io/nba/v1/`) provides REST endpoints covering 1946–present at tiered pricing. The free tier ($0) allows 5 requests/minute but only exposes teams, players, and games. The **GOAT tier** ($39.99/month, 600 req/min) unlocks advanced stats, play-by-play, box scores, season averages, standings, contracts, and betting odds. Authentication uses an API key in the `Authorization` header. Cursor-based pagination returns max 100 records per page. Python and JavaScript SDKs are available (`pip install balldontlie`).

### SportRadar is the enterprise option with live data

As the **official NBA data partner** since 2017, SportRadar provides sub-second real-time play-by-play, but historical data only extends to **2013**. Pricing starts around **$500–$1,000+/month** with a 30-day free trial. Its v8 REST API returns JSON/XML with player profiles, schedules, injuries, depth charts, and event-level possession data. The key limitation for a clone project is the **lack of pre-2013 data** — you would still need NBA.com or Kaggle datasets for historical coverage.

### Open datasets fill the historical gap

Several free, high-quality datasets exist:

- **sumitrodatta's Kaggle dataset** ("NBA/ABA/BAA Stats") contains CSV files scraped from Basketball-Reference covering **1947 to present** across per-game, totals, advanced, shooting, play-by-play, and all-star data. This is the single best free historical backfill source.
- **wyattowalsh's "basketball" Kaggle dataset** provides a comprehensive **SQLite database** with players, teams, games, and drafts — ready for direct import.
- **shufinskiy/nba_data** on GitHub offers pre-compiled **Parquet files** of play-by-play data from three sources: stats.nba.com (1996–97+), pbpstats.com (2000–01+), and data.nba.com with court coordinates (2016–17+). This eliminates the hours-long scraping process for PBP data.

### Salary, injury, and spatial data each require dedicated sources

**Salary data** is best obtained from **HoopsHype** (free, scrapable at `hoopshype.com/salaries/`) for historical data, or the **Spotrac API** (paid, enterprise pricing) for structured access. Basketball-Reference's own contract pages (`/contracts/`) cover active contracts. The `py_ball` Python library includes `TeamSalaries` and `HistoricalSalaries` classes that scrape HoopsHype directly.

**Injury data** comes from **Prosports Transactions** (`prosportstransactions.com/basketball/`) for historical records dating back decades, plus the `nbainjuries` Python package for structured official NBA injury reports from 2021–22 onward.

**Shot location / spatial data** is available free through the NBA.com `shotchartdetail` endpoint from **1996–97 onward**, providing `LOC_X` and `LOC_Y` court coordinates in 1/10th-foot units from the basket. **Second Spectrum** (acquired by Genius Sports for $200M in 2021) provides the richest tracking data — real-time 3D positions of all 10 players, the ball, and referees at 25 frames/second — but this is **not publicly available**. What IS public via NBA.com endpoints are aggregated tracking stats: speed, distance, touches, drives, catch-and-shoot, pull-up shooting, and defensive metrics, available from the **2013–14 season** onward.

### How Basketball-Reference obtained pre-digital historical data

The site's historical completeness traces back to **Dick Pfander**, a Department of Defense employee who began collecting NBA box scores as a grammar school student in the late 1940s. Over decades, he searched microfilm archives in libraries across Michigan, Pittsburgh, Philadelphia, Cleveland, Toronto, and Providence. He printed, clipped, and glued microfilm box scores — spending 10–15 hours per week in the 1960s–70s. After retirement in the early 1980s, he systematically tracked down the last missing games, eventually locating the final missing box score (a 1946 BAA game) on microfilm in Lansing, Michigan. His collection of approximately **54,000 games** was digitized by Justin Kubatko of Basketball-Reference around 2008.

## Part 3: Scraping Basketball-Reference.com

### URL patterns follow predictable conventions

Basketball-Reference uses clean, deterministic URL structures. Every URL can be constructed programmatically:

| Player shooting | `/players/{letter}/{id}/shooting/{year}` | `/players/j/jamesle01/shooting/2024` |

### The robots.txt and rate-limiting rules

Basketball-Reference's `robots.txt` explicitly **disallows** several path patterns for all bots:

```
User-agent: *
Disallow: /play-index/*.cgi?*
Disallow: /play-index/plus/*.cgi?*
Disallow: */gamelog/
Disallow: */splits/
Disallow: */on-off/
Disallow: */lineups/
Disallow: */shooting/
Crawl-delay: 3
```

This means game logs, splits, on-off data, lineups, and shooting pages are **explicitly off-limits to bots** per robots.txt. The Stathead search tools (`/play-index/`) are also disallowed. However, player career pages, team season pages, league summary pages, box scores, draft pages, and standings **are allowed**.

The rate limit is enforced at **20 requests per minute** (3 seconds between requests). Exceeding this triggers a **429 response** and jails your session for up to one hour. Sports Reference's terms of service explicitly prohibit creating competing databases or using scraped data for AI training.

### Comment-wrapped tables are the central parsing challenge

Basketball-Reference's most important quirk: **most tables beyond the first on any page are wrapped in HTML comments** (`<!-- ... -->`) for performance. A standard HTML parser or `pandas.read_html()` call will miss these tables entirely. The `advanced`, `totals`, `per_minute`, `per_poss`, `shooting`, and `pbp` tables on player career pages are all comment-wrapped.

The solution requires BeautifulSoup to extract comment nodes and re-parse them:

```python
import requests, pandas as pd, time
from bs4 import BeautifulSoup, Comment

def extract_bref_table(url, table_id, delay=3.5):
    """
    Extract any table from Basketball-Reference, including
    those hidden inside HTML comments.
    """
    time.sleep(delay)
    headers = {'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36'}
    resp = requests.get(url, headers=headers)
    resp.raise_for_status()
    soup = BeautifulSoup(resp.content, 'html.parser')

    # First, try finding the table directly in visible HTML
    table = soup.find('table', {'id': table_id})

    # If not found, search inside HTML comments
    if table is None:
        for comment in soup.find_all(string=lambda t: isinstance(t, Comment)):
            if table_id in str(comment):
                comment_soup = BeautifulSoup(str(comment), 'html.parser')
                table = comment_soup.find('table', {'id': table_id})
                if table:
                    break

    if table is None:
        raise ValueError(f"Table '{table_id}' not found at {url}")

    # Remove over_header and repeated thead rows before parsing
    for tr in table.find_all('tr', class_=['over_header', 'thead']):
        tr.decompose()

    df = pd.read_html(str(table))[0]

    # Handle MultiIndex columns from surviving multi-level headers
    if isinstance(df.columns, pd.MultiIndex):
        df.columns = [
            '_'.join(str(c) for c in col if 'Unnamed' not in str(c)).strip('_')
            for col in df.columns
        ]

    # Remove repeated header rows that appear every 20th row
    if 'Rk' in df.columns:
        df = df[df['Rk'] != 'Rk'].dropna(subset=['Rk'])

    return df
```

### Table IDs map directly to Basketball-Reference's stat categories

Basketball-Reference recently **renamed some table IDs** (breaking older scrapers). The current mapping:

| Stat type | Old table ID | Current table ID |
|---|---|---|
| Per game | `per_game` | `per_game_stats` |
| Totals | `totals` | `totals_stats` |
| Per 36 min | `per_minute` | `per_minute_stats` |
| Per 100 poss | `per_poss` | `per_poss_stats` |
| Advanced | `advanced` | `advanced` (unchanged) |
| Shooting | `shooting` | `shooting` |
| Play-by-play | `pbp` | `pbp` |
| Salaries | `all_salaries` | `all_salaries` |
| Standings (East) | `divs_standings_E` | `divs_standings_E` |
| Standings (West) | `divs_standings_W` | `divs_standings_W` |
| Draft | `stats` | `stats` |

On season-wide pages (`/leagues/NBA_2024_per_game.html`), the primary table uses `per_game_stats`. On player pages, the career per-game table is `per_game`.

### The over_header problem in multi-level column tables

Many Basketball-Reference tables use **two-level column headers**. The top row (class `over_header`) groups columns — for example, "Shooting" spanning FG, FGA, FG%, and "Per Game" spanning various rate stats. Beneath it is the actual column-name row. Below both, Basketball-Reference repeats the header rows every **20 rows** for readability (class `thead`). These repeated rows must be removed or they become data rows in your DataFrame.

Three CSS classes must be stripped: `over_header`, `over_header thead`, and `thead`. The code above handles this with `tr.decompose()` before calling `pd.read_html()`.

### A production-grade scraper class

```python
import requests, pandas as pd, time, random
from bs4 import BeautifulSoup, Comment
from requests.exceptions import RequestException

class BRefScraper:
    BASE = "https://www.basketball-reference.com"

    def __init__(self, delay=3.5):
        self.delay = delay
        self.session = requests.Session()
        self.session.headers.update({
            'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36'
        })
        self._last_req = 0

    def _wait(self):
        elapsed = time.time() - self._last_req
        if elapsed < self.delay:
            time.sleep(self.delay - elapsed + random.uniform(0, 0.5))
        self._last_req = time.time()

    def _get(self, path, retries=3):
        url = f"{self.BASE}{path}"
        for attempt in range(retries):
            self._wait()
            try:
                r = self.session.get(url, timeout=30)
                if r.status_code == 429:
                    time.sleep(60 * (attempt + 1))  # Exponential jail wait
                    continue
                r.raise_for_status()
                return r.content
            except RequestException:
                time.sleep(5 * (attempt + 1))
        raise Exception(f"Failed to fetch {url}")

    def _table(self, html, table_id):
        soup = BeautifulSoup(html, 'html.parser')
        tbl = soup.find('table', id=table_id)
        if not tbl:
            for c in soup.find_all(string=lambda t: isinstance(t, Comment)):
                if table_id in str(c):
                    tbl = BeautifulSoup(str(c), 'html.parser').find('table', id=table_id)
                    if tbl: break
        if not tbl:
            return None
        for tr in tbl.find_all('tr', class_=['over_header', 'thead']):
            tr.decompose()
        df = pd.read_html(str(tbl))[0]
        if isinstance(df.columns, pd.MultiIndex):
            df.columns = ['_'.join(str(c) for c in col if 'Unnamed' not in str(c)).strip('_')
                          for col in df.columns]
        return df

    def player_career(self, pid, stat='per_game'):
        html = self._get(f"/players/{pid[0]}/{pid}.html")
        return self._table(html, stat)

    def season_leaders(self, year, stat='per_game'):
        html = self._get(f"/leagues/NBA_{year}_{stat}.html")
        tbl_map = {'per_game': 'per_game_stats', 'totals': 'totals_stats',
                    'advanced': 'advanced_stats', 'per_minute': 'per_minute_stats',
                    'per_poss': 'per_poss_stats'}
        return self._table(html, tbl_map.get(stat, f'{stat}_stats'))

    def standings(self, year):
        html = self._get(f"/leagues/NBA_{year}.html")
        return self._table(html, 'divs_standings_E'), self._table(html, 'divs_standings_W')

    def draft(self, year):
        html = self._get(f"/draft/NBA_{year}.html")
        return self._table(html, 'stats')

    def box_score(self, date_str, home_abbrev):
        """date_str format: '20231225', home_abbrev: 'LAL'"""
        html = self._get(f"/boxscores/{date_str}0{home_abbrev}.html")
        basic = self._table(html, f'box-{home_abbrev}-game-basic')
        adv = self._table(html, f'box-{home_abbrev}-game-advanced')
        return basic, adv
```

### Known gotchas and edge cases

**ABA/BAA data**: Players who played in the ABA (1967–1976) or BAA (1946–1949) have stats on Basketball-Reference but use different league identifiers. The `lg` column distinguishes `NBA`, `ABA`, and `BAA`. Combined stats for players who played in both leagues (e.g., Julius Erving) require special handling.

**Player pages with no tables**: Some very early players (pre-1950) have minimal pages with only biographical data and no statistical tables. Your scraper must handle `None` returns gracefully.

**Box score table IDs include the team abbreviation**: On game pages, player box score tables use IDs like `box-LAL-game-basic` and `box-BOS-game-advanced`. You must know both teams to extract all data from a game page.

**The `data-stat` attribute**: Every `<td>` in Basketball-Reference tables has a `data-stat` attribute with the stat's programmatic name (e.g., `data-stat="pts_per_g"`, `data-stat="trb_per_g"`). This is more reliable than column position for extraction and is a Scrapy CSS selector's best friend: `row.css('td[data-stat="pts_per_g"]::text').get()`.

**Season year convention**: Basketball-Reference uses the **ending year** of a season. The 2023–24 season is referenced as `2024` in URLs. The `nba_api` library uses the format `'2023-24'`. Balldontlie uses the starting year. These discrepancies cause bugs if not normalized early.

## The recommended architecture for a complete clone

The fastest path to a complete clone combines multiple sources rather than relying on any single one. Start with **Kaggle datasets** (sumitrodatta's 1947–present CSV files) for the full historical backfill — this gives you player stats, team data, and season totals in hours rather than weeks. Layer in **shufinskiy/nba_data** Parquet files for play-by-play (1996+) and shot details. Use **nba_api** for ongoing daily updates, player biographical data (`CommonPlayerInfo`), and shot chart coordinates (`ShotChartDetail`). Fill salary data from HoopsHype scraping. Reserve direct Basketball-Reference scraping for gap-filling and validation.

Store raw totals and compute derived views (per-game, per-36, per-100) to maintain a single source of truth. Compute advanced stats (PER, WS, BPM, VORP) using the formulas published on Basketball-Reference's glossary pages, which require league averages stored in the `league_season` table. This approach gives you full control over the calculation pipeline and the ability to recompute when methods are updated — exactly as Basketball-Reference itself operates.

## Conclusion

Three insights emerge from this research that are not obvious from surface-level planning. First, **the pre-compiled datasets are underappreciated** — downloading sumitrodatta's Kaggle CSVs and shufinskiy's Parquet files eliminates months of scraping work and avoids the ethical and legal gray areas of large-scale scraping. Second, **the NBA.com API's cloud-IP blocking is the single biggest operational headache** for deployed applications; a residential proxy service (not a cloud proxy) is a practical necessity. Third, **schema design matters more than data collection** — the decision to store raw totals and compute views, handle multi-team seasons with explicit TOT rows, and use nullable columns for era-appropriate gaps determines whether the system degrades gracefully across 80 years of evolving stat-keeping or breaks on edge cases. The table IDs, URL patterns, and code examples above provide the concrete building blocks; the architecture decisions determine whether they assemble into something maintainable.
